Views

self_service_kiosk.views.account_details(request)[source]

Displays a comprehensive summary of the logged-in user’s account details, including orders and payments made within the current year. This view is restricted to non-guest users, ensuring that sensitive financial data remains confidential.

  • Filtering orders and payments specific to the user and the current year.

  • Calculating the total unpaid amount across all orders.

  • Determining if the total amount reflects a debit or credit balance.

The view also adapts its response based on whether the request is coming from a WebView, enhancing compatibility with mobile apps or embedded browsers.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object rendering the account details page with order and payment information, or a simple HttpResponse denying access for guest users.

self_service_kiosk.views.all_products_view(request)[source]

Loads and displays all Category objects along with their associated MenuItem objects for users, differentiated by the “event_user”. Categories and menu items are displayed based on user type and category event settings.

For a normal user: - Only categories where event_category is set to False are displayed.

For the “event_user”: - Displays all categories, including those where event_category is set to True.

Hidden MenuItem objects (where hidden is True) are excluded from the display for all users.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object rendering the all products page with dynamic content based on the user type and category settings.

self_service_kiosk.views.cart(request)[source]

Displays the current user’s shopping cart. This view provides a detailed list of MenuItem in the cart, including their quantities, prices, and the subtotal for each item. It calculates the total price of all items in the cart and allows users to adjust item quantities through POST requests.

  • The cart contents are displayed with an option to update quantities or proceed to checkout.

  • Adjustments made via POST requests update the cart session data and the view is refreshed to reflect these changes.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object rendering the cart page with the possibility to modify quantities or proceed to checkout.

self_service_kiosk.views.confirm_order(request)[source]

Processes the final step in confirming an order based on the presence of payment details. If payment details (like a transaction ID) are provided and validated, an Order is created and marked as paid. Otherwise, the order is created but marked as not paid.

  • The function checks for a “paid” query parameter in the request to determine if payment has been processed.

  • An Order instance is created with a status reflecting the payment state.

  • On successful order creation, the user is redirected to a success page with details about the order status.

  • In the absence of payment confirmation, the order is still processed but marked as pending, and appropriate actions are taken based on the business logic.

Parameters:

request – The HTTP request object.

Returns:

Redirects to a success or status page detailing the order outcome.

self_service_kiosk.views.forgot_password(request)[source]

Allows users to request a password reset. On GET, it displays the password reset form. On POST, it verifies the existence of a user by username and sends a password reset link if the user is active.

  • The link is generated using a PasswordResetToken which is sent to the user’s registered email.

  • Appropriate success or error messages are displayed based on the user’s existence and their account status.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object rendering the password reset page, or redirecting with a status message.

self_service_kiosk.views.guest_login(request)[source]

Provides a login mechanism for a guest user with a hardcoded username. This function checks if the hardcoded guest user exists and logs them in, redirecting to the all products page. If the guest user cannot be authenticated, an error message is displayed.

  • This is intended for limited functionality access without creating a personal account.

Parameters:

request – The HTTP request object.

Returns:

A redirect to the all products page on successful login, or the login page with an error message on failure.

self_service_kiosk.views.help_page(request)[source]

Renders a help page where users can submit feedback via a form. This POST endpoint handles the form submission, validates the feedback, and sends it via email if valid. A confirmation message is displayed after successful submission.

  • Enhances user support by providing a direct method for feedback submission and immediate response handling.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object either displaying the form with success confirmation or re-rendering the form for further input.

self_service_kiosk.views.invoice_view(request, file_path)[source]

Securely delivers invoice files to authorized users. Validates if the requested file exists and checks if the user has the appropriate permissions to access it. This view handles the display or download of invoices stored within a secure directory.

  • Ensures that access to invoice files is restricted to authorized personnel, preventing unauthorized data access.

Parameters:

  • request – The HTTP request object.

  • file_path – The relative path to the invoice file intended for access.

Returns:

A FileResponse allowing the authorized download of the invoice, or an HttpResponseForbidden if access is denied.

self_service_kiosk.views.login_with_token(request, token)[source]

Facilitates user login using a secure token, typically sent to the user via email or embedded in a QR code. Validates the token and logs the user in if the token is valid and not expired.

  • This provides an alternative login method that can be more secure and convenient for users, especially for single-use or time-sensitive sessions.

Parameters:

  • request – The HTTP request object.

  • token – The token used for the user login attempt.

Returns:

A redirect to the main index page on successful login or back to the login page with an error message on token validation failure.

self_service_kiosk.views.order_detail(request, order_id)[source]

Displays detailed information about an order specified by the order_id. This view checks if the logged-in user has the appropriate permissions to view the order details. The view includes comprehensive information about the order including items, quantities, and prices at the time of purchase.

  • Provides a detailed breakdown of the order to support customer service and order management operations.

Parameters:

  • request – The HTTP request object.

  • order_id – The identifier of the order to be detailed.

Returns:

An HttpResponse object rendering the detailed view of the order if access is permitted, or an HttpResponseForbidden if the user is not authorized to view the details.

self_service_kiosk.views.order_success(request, case)[source]

Displays a success message to the user after an order has been processed, based on the provided case parameter which indicates the specific scenario of the order process. The function handles different cases:

  • Case 1: Order was successfully placed but was not paid directly or fully.

  • Case 2: Order was placed and payment was successfully processed.

  • Other cases may represent various error or informational states related to order processing.

The appropriate message is displayed to the user based on the case parameter, providing feedback on the order status.

Parameters:

  • request – The HTTP request object.

  • case – An integer representing the outcome scenario of the order process.

Returns:

An HttpResponse object rendering the order success page with a contextual message based on the case.

self_service_kiosk.views.payment_problem(request)[source]

Handles errors during payment processing by interpreting a specific error code passed as a query parameter. This view renders a detailed error message explaining the potential cause and suggesting next steps or remedies.

  • Each error code corresponds to a particular payment issue, which is explained to the user.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object rendering a page that describes the payment error based on the provided code.

self_service_kiosk.views.process_payment(request)[source]

Manages the post-payment processing by verifying the payment details such as transaction ID and payment type. Based on the verification, it either completes an order or credits an account, handling both “order” and “credit” payment types.

  • Ensures that payments are processed securely and correctly, updates order status, and handles account credits.

Parameters:

request – The HTTP request object.

Returns:

A redirect to appropriate views depending on the payment outcome and type.

self_service_kiosk.views.qr_code_view(request)[source]

Processes a QR code value submitted via POST. Depending on the prefix in the QR code, it redirects to different functionalities such as logging in, resetting a password, or handling user registration.

  • This method ensures flexibility and user-friendly interactions by using QR codes to initiate complex workflows with simple scans.

Parameters:

request – The HTTP request object.

Returns:

A redirect based on the decoded action from the QR code or a JsonResponse indicating an error if the QR code action is invalid.

self_service_kiosk.views.register(request)[source]

Manages user registration through both GET and POST requests. On a GET request, it renders the registration form. On a POST request, it processes the provided user data to create a new user account and associated RegistrationToken. Upon successful creation, a confirmation email is sent.

  • The email is sent using the RegistrationToken if the account is successfully created.

  • Appropriate success or error messages are displayed based on the outcome.

  • The user is redirected to the login page after registration or back to the registration form if errors occur.

Parameters:

request – The HTTP request object.

Returns:

An HttpResponse object either rendering the registration page or redirecting to the login page with a message.

self_service_kiosk.views.send_feedback(request)[source]

Receives feedback as a JSON payload via a POST request, processes it, and forwards the feedback via email. This function validates the JSON structure, extracts feedback details, and uses an email service to send the feedback to a specified recipient.

  • Facilitates asynchronous feedback collection from users, enhancing user engagement and support.

Parameters:

request – The HTTP request object.

Returns:

A JsonResponse indicating the success or failure of the feedback submission.

self_service_kiosk.views.set_password(request, token)[source]

Allows a user to set or reset their password using a valid token received via email. This view supports POST requests for form submission of the new password and uses Django’s SetPasswordForm for validation.

  • The token’s validity is checked to ensure it’s not expired and matches the user.

  • Upon successful password update, the token is invalidated to prevent reuse, and the user is redirected to the login page.

Parameters:

  • request – The HTTP request object.

  • token – The token used for identifying the valid password reset request.

Returns:

A redirect to the login page upon successful password reset, or the same form with error messages if unsuccessful.

self_service_kiosk.views.update_quantity(request, item_id, quantity)[source]

Dynamically updates the quantity of a specific MenuItem in the user’s cart via an AJAX call. This view handles quantity updates on the client side, immediately reflecting changes without reloading the page.

  • Validates the new quantity and updates the cart session data accordingly.

  • If the quantity is set to zero, the item is removed from the cart.

  • Returns a JsonResponse indicating the success or failure of the update, including the new quantity or a removal confirmation.

Parameters:

  • request – The HTTP request object.

  • item_id – The ID of the MenuItem whose quantity is to be updated.

  • quantity – The new quantity to be set; if zero, the item is removed.

Returns:

A JsonResponse indicating the outcome of the operation.

Handles user authentication with a username and password submitted through POST. If successful, the user is logged in and redirected to the all products page. If unsuccessful, the login page is re-rendered with an error message.

  • If a GET request is detected with credentials, those are also used to attempt a login.

  • Utilizes Django’s built-in authenticate and login functions to manage user sessions and security.

Parameters:

request – The HTTP request object.

Returns:

A redirect to the all products page upon successful login, or the login page with an error message on failure.

Handles the logout process for the current user. This function terminates the user session effectively logging out the user, and redirects them to the login page. This ensures that all session data is cleared and the user must re-authenticate to access protected resources.

Parameters:

request – The HTTP request object.

Returns:

A redirect to the login page, ensuring the user session is cleanly terminated.

Unit Tests

In this chapeter we will explain how to run the unit tests for the Self-Service-Kiosk Django application.

The test concept for the Self-Service-Kiosk Django application is focused the views of the application. Each view has a corresponding test class in the self_service_kiosk/tests directory and is tested with multiple test cases. As a result the code coverage of views.py is 100%.

The tests are written using the Django testing framework and are located in the self_service_kiosk/tests directory.

Run Unit Tests

To simply run all unit tests, execute the following command in the parent mysite directory:

python manage.py test

Run Tests with Coverage

To run all unit tests and get a coverage html report you need to first install the coverage Python package:

pip install coverage

Then you can run the unit tests with coverage and generate the html report with the following commands:

coverage run --source='.' manage.py test
coverage html

After executing the above commands, you can find the coverage html report in the htmlcov directory.

Building this documentation

If you want to build this documentation for yourself, you need to install the sphinx Python package and the spinx extensions used for this project:

pip install sphinx sphinx_rtd_theme sphinxcontrib_django

Then you can build the documentation by executing the following command in the mysite directory:

sphinx-build -b html source build

After executing the above command, you can find the built documentation in the build directory.
